# Твоя первая DevOps Практика (Проект FastAPI)

Добро пожаловать! Это твой тренировочный полигон. У нас уже есть готовое веб-приложение (API), написанное на языке Python с использованием фреймворка FastAPI. Но сам по себе современный код — это просто файлы кодом. Чтобы приложение работало в интернете, его нужно правильно запускать, подключать к базам данных и автоматически тестировать.

Этим и занимаются DevOps-инженеры. Твоя миссия: заставить этот код работать везде ("на любой машине"), а не только на компьютере разработчика.

### Что нужно знать перед стартом:
- **Docker**: Программа, которая позволяет упаковывать твое приложение вместе со всеми его библиотеками в единую "коробку", чтобы оно запускалось одинаково на любой операционной системе (Windows, Mac, Linux).
- **Образ (Image)**: Это "чертеж" или "рецепт" твоей коробки с приложением. Ты пишешь инструкции текстом в файле под названием `Dockerfile`, и на базе этого текста Docker собирает образ.
- **Контейнер (Container)**: Это "оживший" образ. Представь, что это запущенный крошечный невидимый виртуальный компьютер внутри твоего реального компьютера. На нём нет рабочего стола, на нём работает только одна единственная программа (твоё API или база данных) и больше вообще ничего.
- **CI/CD**: Идея непрерывной работы с кодом. Если коротко: робот (например, GitHub Actions) автоматически проверяет твой код (запускает тесты), как только ты заливаешь его на GitHub.

> 🛠 **Как запускать тесты локально?**
> В репозитории уже есть файл `Makefile`. В нем прописаны полезные команды-сокращения. Прежде чем паковать всё в Docker, убедись, что код вообще работает. Для проверки правильности кода (линтеры) и авто-тестов разработчиков используй в терминале команду:
> `make test` или `make verify`
> Изучай `Makefile` подглядывая в него — там много полезного!

## Этап 1: Одиночный Docker-контейнер для API

Для начала нам нужно упаковать наше API в Docker-образ. Без Docker тебе пришлось бы просить друзей вручную устанавливать на их Windows нужную версию Python, настраивать виртуальные окружения и качать библиотеки. С Docker достаточно одной команды.

В проекте для установки пакетов Python используется супер-быстрый инструмент `uv` (современный аналог `pip` или `npm`).

**Твои шаги:**
1. Создай новый файл с точным именем `Dockerfile` в самой главной (корневой) папке проекта. У него не должно быть расширения `.txt`.
2. Загугли базовые инструкции (команды) для Докера: `FROM`, `WORKDIR`, `COPY`, `RUN`, `CMD`, `ENV`.
3. Напиши внутри файла инструкции, чтобы Docker сделал следующее:
   - Взял за отправную точку базовый образ Python из интернета (например, `python:3.12-slim`).
   - Установил зависимости приложения (Docker должен прочитать файлы `pyproject.toml` и `uv.lock`). *Подсказка: продвинутый пример того, как быстро установить зависимости через `uv` внутри образа, можно подсмотреть в шпаргалках `www/dockerfile_example` и `www/dockerfile_example_2` в этом же проекте.*
   - Скопировал исходный код проекта с твоего диска внутрь образа.
   - Запустил веб-сервер при старте контейнера (тут полезна команда `make start`, посмотри что она внутри вызывает).
4. Создай файл `.dockerignore` (работает как `.gitignore`). Добавь туда папки `.venv`, `.git/`, `__pycache__`, потому что их копирование внутрь образа ни к чему хорошему не приведёт и сделает сборку долгой.
5. Собери образ через команду в терминале: `docker build -t my-app .` (точка в конце очень важна!).
6. Запусти из него контейнер: `docker run -p 8000:8000 my-app` и попытайся открыть `http://localhost:8000/docs` в браузере.

## Этап 2: Оркестрация через Docker Compose

Ура, само API работает! Но реальным приложениям нужна ещё и база данных (например, PostgreSQL), чтобы хранить регистрации пользователей и другую информацию.
Неудобно запускать базу данных, потом запускать API, связывать их длиннющими командами по сети... Для этого придумали **Docker Compose** — инструмент, который запускает сразу целый "оркестр" разных контейнеров по одному понятному файлу-конфигурации (`.yml`).

**Твои шаги:**
1. Создай файл `docker-compose.yml`. *Подсказка: посмотри на пример синтаксиса в файле `www/docker_compose_example`.*
2. Добавь туда два сервиса (services): 
   - `api` — твое приложение. Укажи в настройках сервиса, что его образ нужно собирать (build) из `Dockerfile`, который ты написал на Этапе 1.
   - `db` — база данных. Используй официальный готовый образ `postgres` из Докерхаба. База данных просто так не включится: ей обязательно нужны пароли и логины администратора. Укажи их через переменные окружения (`environment`).
3. **Зависимости:** Твоё API упадёт с ошибкой, если попытается подключиться к БД, которая ещё грузится. Сделай так, чтобы сервис `api` стартовал только **после** запуска `db` (найди в документации параметр `depends_on`).
4. **Volumes (Тома):** Контейнеры не хранят данные вечно. Если ты удалишь контейнер БД, все записи в таблицах исчезнут. Чтобы база данных сохраняла информацию "навсегда", подключи к ней `volumes` (раздел на твоём жестком диске).
5. (Дополнительно) Для сервиса `api` тоже настрой монтирование `volumes` (привяжи папку `src` с твоего компа внутрь папки приложения в контейнере). Зачем? Так сервер будет сразу видеть, когда ты меняешь код в редакторе, и сам перезагружаться без пересборки (hot-reload)!
6. Запусти всю инфрастуктуру одной командой: `docker-compose up`.

## Этап 3: Роботы-проверяющие (CI/CD в GitHub Actions)

Когда ты работаешь в команде, люди постоянно отправляют новый код. Кто знает, сломал ли Вася своими изменениями логику приложения или нет? Нам нужен проверяющий робот.
GitHub разрешает бесплатно запускать скрипты на их серверах при каждом коммите (Это называется GitHub Actions).

**Твои шаги:**
1. Создай папку `.github` (с точкой), внутри неё папку `workflows`, а внутри неё файл `ci.yml`.
2. Напиши синтаксис (поищи в интернете "GitHub Actions python workflow template"). Скрипт должен срабатывать (trigger / on) автоматически при событии `push` в главную ветку или при создании `Pull Request`.
3. Укажи операционную систему для робота: `ubuntu-latest`.
4. **Шаги (Steps), которые робот обязан выполнить по порядку:**
   - Скачать код репозитория (используй готовый action `actions/checkout`).
   - Установить Python.
   - Установить зависимости репозитория (через `uv` или просто `pip install`).
   - Запустить линтер (доктор-проверяющий код на опечатки и пробелы). Вбить команду: `make lint` или `make format`.
   - Запустить тесты API. Вбить команду: `make test`.
5. Как только ты сделаешь `git push`, зайди на GitHub в раздел "Actions" своего репозитория и полюбуйся, как работает магия (или почему она упала красным крестиком).

## Этап 4: Подготовка к продакшену (Для продвинутых)

Образ Docker, который ты сделал в Этапе 1, идеален для разработки. В нём лежат утилиты для тестирования, линтеры, разные компиляторы, и он весит очень много гигабайт. Но на боевой сервер в дата-центре (Production / Прод) мы отправляем только самый минимум. 

**Твои шаги:**
1. Создай отдельный файл конфигурации `docker-compose.prod.yml`. Здесь уже НЕ НУЖНО монтировать папку с кодом в контейнер. Образ должен быть запечатанным и содержать код внутри себя.
2. Открой свой `Dockerfile` и используй трюк **Multi-stage build** (Многослойная сборка). 
   - Смысл в том, чтобы сделать сборку в 2 этапа "в самом файле":
   - Этап-строитель (builder): Скачивает все зависимости, ставит тяжелые компиляторы и подготавливает файлы.
   - Этап-доставки (runtime): Берет абсолютно чистый образ Python, копирует _только_ установленные в первом этапе готовые библиотеки `venv` (и сам код). И больше ничего лишнего. *Техника Multi-stage описана в примерах `www/dockerfile_example` и `www/dockerfile_example_2`.*
3. Добавь в продакшен версию compose-файла политику перезапуска баз данных и API: `restart: always` (если в дата-центре моргнет свет и сервер перезагрузится, Docker автоматически поднимет твои контейнеры сам).

## 🆘 Советы для выживания (если ты застрял)

1. **Docker жалуется на порты?**
   > Ошибка `port is already allocated` значит, что на твоем ПК порт `8000` (или `5432` для базы) уже чем-то занят. Может ты забыл остановить локальный сервер `make start` или старый контейнер? Проверь через `docker ps`.
2. **Как удалить мусор из Docker?**
   > Если наплодилось куча сломанных образов и контейнеров, используй команды пылесоса: `docker system prune -a` или `docker rm -f $(docker ps -aq)`. Осторожно, удалит все локальные образы!
3. **Как попасть ВНУТРЬ запущенного контейнера?**
   > Ищешь куда пропал файл? Введи `docker exec -it <имя_или_ID_контейнера> /bin/bash` (или `/bin/sh`) и ты окажешься прямо в терминале сервера внутри контейнера!
4. **Контейнеры не видят друг друга внутри Docker Compose?**
   > В Docker Compose контейнеры общаются **НЕ** по `localhost`. Они общаются по названиям сервисов! То есть, если в `.yml` база называется `db`, то API должно подключаться к хосту: `postgresql://user:pass@db:5432/dbname`. Вместо IP адреса пиши просто `db`.
5. **Тесты в GitHub Actions падают, а локально работают?**
   > Скорее всего ты забыл поднять базу данных для тестов внутри GitHub Actions, или тесты пытаются подключиться к базе на твоём компьютере. GitHub Action — это чистый сервер, там нет запущенного postgres, его надо туда тоже добавить!
